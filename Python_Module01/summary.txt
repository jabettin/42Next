Python Architecture & OOP - Learning Summary
============================================

--- IMPORTS ---
- Don't include .py in import statements
- Cross-directory imports require __init__.py files or sys.path manipulation
- For 42 exercises, "reuse" means copy the class into the new file — no cross-directory imports needed

--- THE DOT OPERATOR ---
- In Python, everything is an object, so the . operator works uniformly on everything
- rose.name → READ: looks up 'name' on rose
- rose.name = "Rose" → WRITE: stores 'name' in rose's __dict__
- Same lookup chain regardless of whether you're accessing data or calling a method
- This is why Python is slower than C — even rose.height requires two pointer dereferences and a dict lookup

--- LOOKUP ORDER ---
- When you access rose.name, Python checks:
  1. Instance __dict__ first
  2. Then the class
  3. Then parent classes
- Instance always takes precedence over the class
- This is why name collision between attributes and methods is dangerous

--- THE age() BUG ---
- rose.age = 8 writes 'age' into rose's __dict__
- Later, rose.age() looks up 'age' in __dict__ first, finds the integer 8
- Python tries to call 8 as a function → "int object is not callable"
- The overwrite happens on the ASSIGNMENT line, not at the call site
- Fix: use a different name for the attribute (days_old) so it doesn't shadow the method

--- MEMORY MODEL ---
Stack:
  rose → 0x1A2B (just a pointer, 8 bytes)

Heap:
  0x1A2B (instance)
  ├── __class__ → 0x3C4D  (pointer to Plant class)
  └── __dict__  → { name: "Rose", height: 10.0, days_old: 30 }

  0x3C4D (Plant class, created once, lives for program lifetime)
  ├── show  → pointer to function
  ├── grow  → pointer to function
  └── age   → pointer to function

- rose is just a pointer on the stack — all data lives on the heap
- The class is like a vtable in C, shared by all instances
- Multiple variables can point to the same instance (like assigning pointers in C)

--- SELF ---
- self is just the instance pointer passed as the first argument to every method
- rose.show() is identical to Plant.show(rose) under the hood
- self.name and rose.name are the same memory location when called on rose
- Setting self.name = "x" inside a method has the same effect as rose.name = "x" from outside

--- INSTANTIATION ---
- rose = Plant() is a function call that:
  1. Allocates a new instance on the heap
  2. Calls __init__(instance, ...) to set up data
  3. Returns the address of the instance
  4. rose now holds that address
- rose is the receiver of the result, not an input
- rose = None first, then rose = Plant() is valid — just reassigning a pointer

--- __init__ ---
- Bridges temporary stack parameters to permanent heap instance attributes
- local parameter 'name' lives on the stack and dies when __init__ returns
- self.name lives on the heap instance and survives
- Without __init__, attributes must be set manually after instantiation

--- CLASS-LEVEL DECLARATIONS ---
- class Plant:
      name: str
      height: float
      days_old: int
- These are purely for mypy — they do nothing at runtime
- Mypy treats them as declarations that instances will have these attributes
- The actual values still come from assignment (rose.name = "Rose" or self.name = ... in __init__)
- name: str on the class and self.name on the instance are separate,
  but mypy maps class-level annotations to instance attributes as a convention

--- TYPE HINTS ---
- Methods: def show(self) -> None
- Variables: rose: Plant | None = None
- Plant | None means the variable can hold a Plant or None (Python 3.10+ syntax)
- Don't annotate external attribute assignments: rose.name = "Rose" (not rose.name: str = "Rose")
- mypy checks types statically without running the code

--- FLAKE8 RULES (PEP8) ---
- Use 4 spaces for indentation, not tabs (W191)
- Blank line between methods in a class (E301)
- 2 blank lines after class definition before if __name__ (E305)
- Spaces around operators: def show(self) -> None not def show(self)->None (E225)
- Newline at end of file (W292)

--- SHEBANG ---
- #!/usr/bin/env python3 as the first line of a file
- Needed when running the file directly: ./ft_plant_growth.py
- Not needed when running with: python3 ft_plant_growth.py
- /usr/bin/env python3 finds python3 via PATH (more portable than hardcoding the path)
- Only necessary for interpreted languages — compiled languages produce executables the OS runs directly
- chmod +x ft_plant_growth.py makes the file executable

--- RUNNING TOOLS ---
- python3 -m mypy ft_plant_growth.py    (type checking)
- python3 -m flake8 ft_plant_growth.py  (style linting)
- Same workflow as norminette in C
